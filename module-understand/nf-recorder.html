NextFrame 模块总览 #01

录制器 nf-recorder

把一个网页变成一个视频文件。
不是录屏，是逐帧拍照再装订成册。

📷

想象一个超高速照相机

你有一个 HTML 网页，里面有动画、有文字、有特效。普通做法是播放这个网页然后录屏——但录屏会丢帧、画质差。

recorder 的做法不同：它在内存里打开一个看不见的浏览器，然后像定格动画一样—— "时间暂停 → 拍一张照 → 时间走一格 → 再拍一张"，如此重复几百上千次，最后把所有照片合成视频。

所以不管动画多复杂，每一帧都是完美的。就像用相机一张张拍，而不是用手机屏幕录制。

这个模块做什么
一进一出，非常纯粹
recorder 是一个独立的转换器。不管 HTML 从哪来、谁生成的，只要给它一个网页，它就给你一个视频。
输入
🌐

HTML 文件

一个自包含的网页文件
里面有引擎代码，能在任意时刻渲染画面
可能还带音频路径、视频图层信息

nf-recorder
输出
🎬

MP4 视频

H.264 编码的标准视频文件
音画已合并，可直接上传各平台
附带性能日志（录了几帧、多快、多大）

怎么做到的
7 步流水线
从拿到 HTML 到输出 MP4，中间经过 7 个工序。橙色高亮的是核心步骤。
📋

1. 收集文件

找到要录制的 HTML 文件。可以给一个文件，也可以给一个文件夹（里面的 HTML 按顺序录制后拼成一个视频）。

🔍

2. 读取信息

打开 HTML 文件，从里面提取关键信息：这个视频多长？有没有配音？有没有字幕？有没有需要叠加的真人视频？这些信息都嵌在 HTML 的代码里。

🧮

3. 制定拍摄计划

根据视频时长和帧率，算出一共需要拍多少张照片、每张照片对应哪个时间点。比如一个 10 秒 60fps 的视频 = 600 张照片。

🖥️

4. 打开隐形浏览器

在内存里创建一个看不见的 Safari 浏览器窗口（藏在屏幕外面），把 HTML 加载进去。这个浏览器是真的——能跑 JS、能渲染 CSS、能画 Canvas，和你在 Safari 里打开完全一样。

用的是 macOS 自带的 WKWebView（Safari 内核）
📸

5. 逐帧拍照（核心）

这是最关键的一步。循环几百上千次，每次做同样的事：
① 告诉页面"现在是第 N 帧"（页面收到后会把动画、文字、特效都更新到这个时间点）
② 等页面渲染完
③ 从显卡直接读取画面像素（不是截屏，是读 GPU 的渲染结果）
④ 把这张图交给编码器

优先用 CALayer.render（GPU直读），黑屏则自动换 takeSnapshot
⚡

6. 实时编码

每拍完一张照片，立刻用 Mac 的硬件视频编码芯片把它压缩成 H.264 格式写入文件。不用等全拍完再编码，边拍边压，速度更快。

用的是 Apple VideoToolbox 硬件加速
🎵

7. 合并收尾

把配音音频合进视频、把多段视频拼接成一个文件、如果有真人视频图层就叠加上去。最终产出一个完整的 MP4 文件。

设计决策
为什么这么做
recorder 有几个关键的设计选择，理解了这些就理解了它为什么长这样。
⏸️

时间由我控制

不是"播放视频然后录屏"，而是录制器控制时间的流逝。 它说"现在是 0.5 秒"，页面就渲染 0.5 秒的画面。 好处：不管动画多复杂，永远不丢帧，每帧都完美。 代价：不能录实时内容（比如摄像头），只能录预设好的动画。

🧠

聪明地偷懒

如果一段时间内画面没变化（比如一段文字静止 3 秒），就不重新拍照，直接复用上一张。 它分析字幕切换点来判断"什么时候画面会变"。 一个 60 秒的视频，可能只需要真正拍 200 张而不是 1800 张。

🏭

多条生产线

把视频切成几段，每段用一个独立的浏览器录制，最后拼起来。 比如 4 段并行，速度提升约 3-4 倍。 特别适合录制 4K 这种比较慢的场景。

🔌

完全独立

recorder 不知道也不关心 HTML 是怎么生成的。 它只认一个约定：HTML 里要有 engine.duration（视频多长）和 __onFrame（接收时间信号）。 满足这两点，任何 HTML 都能录。

内部结构
10 个"房间"，各司其职
recorder 像一座小工厂，有 10 个功能区。每个区负责一件事，互不干扰。
🚪 api/ — 总入口
所有录制任务从这里开始。它决定走哪条路线：一台机器从头录到尾（串行），还是开多台机器各录一段（并行）。
串行由 orchestrator.rs 执行，并行由 parallel/ 目录管理（拆帧 → 多进程 → 拼接）
🖥️ webview/ — 隐形浏览器
创建和管理那个看不见的浏览器。负责加载页面、等待渲染完成、截图。是整个 recorder 最核心的部分。
5 个文件：创建窗口 / 截图(两种方式) / 页面导航 / 注入帧状态 / 离屏管理
🔄 record/ — 录制循环
一段视频的完整生命周期：准备（导航到页面、初始化编码器）→ 逐帧循环（截图+编码 × N 次）→ 收尾（关编码器、合音频）。
frame_loop.rs 是那个"拍一张、走一格"的核心循环
💿 encoder/ — 视频编码
把每张截图压缩成视频流，写入 MP4 文件。用 Mac 芯片的硬件加速，比软件编码快很多。
pipeline.rs 管编码写入 / pixel_buffer.rs 管格式转换 / ffmpeg.rs 管音频合并和拼接
📖 parser/ — 信息提取
从 HTML 源码里读取录制所需信息：视频多长？配音文件在哪？有几条字幕？有哪些视频图层？
用正则表达式从 JS 代码里提取变量值（duration、audioSrc、subtitles 等）
📐 plan.rs — 拍摄计划
收集所有要录的 HTML 文件，算出每段视频的时长和帧数，生成一份完整的"拍摄计划书"。
如果有音频，以音频时长为准；没有就用字幕时长；都没有就默认 10 秒
⏱️ clock.rs — 帧时钟
决定第 N 帧对应几秒、这一帧需不需要真的拍照。"聪明偷懒"的核心逻辑就在这里。
分析字幕切换时间点 → 在静止区间自动复用上一帧 → 大幅减少实际截图次数
📷 capture.rs — 截图工具
底层图像处理。把浏览器渲染的画面转成标准图片格式，并且检测"是不是拍到了黑屏"。
黑帧检测用 16×16 网格采样，发现黑屏会自动重试最多 3 次
🌐 server/ — 本地服务器
起一个临时的本地网站服务器。因为浏览器有安全限制，直接打开本地文件会受限，通过 HTTP 服务就没这个问题。
录制开始时自动启动，录制结束后关闭。如果启动失败会降级到 file:// 模式
🎞️ overlay/ — 叠加与日志
两件事：① 把真人视频叠加到录制结果的指定区域。② 记录性能数据（录了多快、文件多大、用了什么编码器）。
视频叠加用 FFmpeg overlay 滤镜，支持精确定位 x/y/w/h

为什么竖屏比横屏快？

竖屏用 --dpr 2 渲染 540×960 再放大到 1080×1920，实际截图面积小。横屏 1920×1080 直接渲染，面积更大。截图面积 = 速度瓶颈。

关系
谁用它、它用谁

📥 谁调用 recorder

nf-bridge（桌面应用点"导出"按钮） 命令行直接运行 nextframe-recorder

📦 recorder 依赖的 macOS 能力

WKWebView — Safari 浏览器引擎 VideoToolbox — 硬件视频编码 CoreGraphics — 图像处理 AppKit — 窗口管理 AVFoundation — 媒体框架

🔧 recorder 依赖的 Rust 库

objc2 — Rust 调 macOS 原生 API 的桥梁 serde — 数据序列化 clap — 命令行参数解析

🔗 外部工具

FFmpeg — 音频合并、片段拼接（需要系统安装）
🧩

recorder 是完全独立的

它不依赖 nf-core（引擎）、不依赖 nf-cli（命令行）、不依赖任何 JS 代码。只要给它一个 HTML 文件，不管这个 HTML 是我们的引擎生成的、还是手写的、还是别的工具生成的——只要满足约定（有 duration + __onFrame），它就能录。这种独立性是刻意设计的。
